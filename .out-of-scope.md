# Out of Scope

Concepts explicitly deferred or rejected. Each `## ` heading is a searchable entry for plan-phase deduplication.
Cross-linked from CONTEXT.md §Out of Scope (top-5 summary view).

---

## Embedded web framework (FastAPI/Flask) in the hello-world starter

**Date:** 2026-05-12
**Reason:** Standard Library First constraint; stdlib `http.server` is sufficient for the hello-world starter. Introducing FastAPI or Flask would pull in a third-party dependency that the template does not need and that forks would have to justify removing.
**Source ADR:** N/A — constraint originates from CONTEXT.md Architectural Constraints ("Standard library first").

---

## Multi-variant template (separate per-type scaffolds)

**Date:** 2026-05-10
**Reason:** Replaced by a single template with a `project_type` prompt; specialised directory stubs deferred until a concrete project demands them. Maintaining separate scaffolds per project type would multiply template surface area without proportionate benefit at the current maturity level.
**Source ADR:** N/A — recorded in CONTEXT.md §Out of Scope on 2026-05-10.

---

## Gemini as primary AI provider

**Date:** 2026-05-09
**Reason:** Claude-first architectural constraint. Gemini is reserved as a future secondary slot. Elevating Gemini to primary would contradict the Claude-first constraint in CONTEXT.md and require re-evaluating all skill prompts.
**Source ADR:** N/A — constraint originates from CONTEXT.md Architectural Constraints ("Claude-first AI").

---

## Separate CLAUDE.md for behavioural rules

**Date:** 2026-05-11
**Reason:** Rejected in ADR-009. `AGENTS.md` is the single standing-instructions file (cross-tool convention). A separate `CLAUDE.md` would split the source of truth and privilege one vendor over others.
**Source ADR:** [ADR-009](docs/adr/ADR-009-context-strategy.md) — Option A rejected; Option B chosen.

---

## Generic Module Map per project_type (separate maps for application / agent / ml)

**Date:** 2026-05-11
**Reason:** Rejected in ADR-009. A single Map at template-time lists only invariant rows; project_type-specific rows are added as the first feature lands. Separate per-type maps would fragment the navigation surface and increase maintenance overhead.
**Source ADR:** [ADR-009](docs/adr/ADR-009-context-strategy.md) — confirmation criterion: "Module Map at template-time is shallow by design".

---

## Codespace as HITL validation surface

**Date:** 2026-05-12
**Reason:** HITL surface is the developer's laptop indefinitely. Persistent Cloud Run Orchestrator Service is the Phase-3 cloud-runtime north star, not Codespaces. Codespaces adds provisioning latency and environment parity risk without advancing the current maturity target.
**Source ADR:** N/A — recorded in CONTEXT.md §Out of Scope on 2026-05-12.

---

## LangGraph (or any agent-framework library) as orchestrator runtime

**Date:** 2026-05-12
**Reason:** Drag too high for a template — forks should not need to learn LangGraph. Patterns (declarative graph, checkpoint, interrupt) are stolen without the dependency. Adding LangGraph would make the template non-trivial to fork and would contradict the "standard library first, minimal dependencies" ethos.
**Source ADR:** N/A — recorded in CONTEXT.md §Out of Scope on 2026-05-12.
